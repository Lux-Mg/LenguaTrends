## Спринт №1 — 17.04 – 01.05.2026

### Этап 1. Подготовительный (17.04 – 19.04)

- Прохождение инструктажа по технике безопасности и противопожарной безопасности на рабочем месте.
- Ознакомление с программой практики, согласование индивидуального задания и календарного графика с научным руководителем (утверждены 18.04.2026).
- Выполнен анализ текущего состояния разрабатываемой программной системы LenguaTrends: выявлены основные направления доработки – расширение корпуса комментариев, верификация качества NLP-моделей на собственном корпусе, улучшение пользовательского интерфейса и интернационализации.

### Этап 2.1. Сбор и анализ материала, анализ литературы (20.04 – 24.04)

- Изучены современные методы мультиязычного анализа тональности пользовательских комментариев.
- Рассмотрены архитектуры трансформерных моделей XLM-RoBERTa (Conneau et al., 2020) и их специализированных версий для задачи анализа тональности в социальных сетях (Barbieri et al., cardiffnlp, 2022).
- Выполнен обзор методов тематического моделирования с использованием библиотеки BERTopic и многоязычных эмбеддингов.
- Проведён анализ альтернативных моделей для сравнительного тестирования: distilled-версия XLM-R (lxyuan/distilbert-base-multilingual-cased-sentiments-student), модель tabularisai/multilingual-sentiment-analysis.
- Изучены особенности и ограничения платформы YouTube и её Data API v3 в контексте сбора мультиязычных комментариев, а также возможности TMDB API для получения метаданных кинопроизведений.

### Этап 2.2. Выполнение исследований по теме задания (27.04 – 01.05)

- Проведена количественная оценка качества предварительно обученной модели XLM-RoBERTa на корпусе комментариев LenguaTrends на трёх языках (русский, английский, испанский):
  - Выполнена ручная разметка 399 комментариев (138 ES, 130 EN, 131 RU) в качестве собственной эталонной выборки.
  - Реализован сравнительный бенчмарк трёх моделей на собственной выборке: модель XLM-RoBERTa (Cardiff) показала F1-macro 0,613 против 0,369 и 0,505 у альтернатив.
  - Дополнительная верификация проведена на внешних размеченных датасетах: mteb/tweet_sentiment_multilingual (ES/EN, ~200 образцов на язык) и MonoHime/ru_sentiment_dataset (RU, 198 образцов). Результаты подтвердили превосходство XLM-RoBERTa Cardiff (accuracy 0,64 ES; 0,69 EN).
- Выполнена верификация работы модуля тематического моделирования BERTopic на расширенном корпусе: подтверждена корректная работа кластеризации и интерпретируемость выделяемых тематических кластеров в разрезе каждого из трёх языков.
- Проведён анализ ограничений YouTube Data API v3 и TMDB API: описаны квоты, стратегии дедупликации комментариев, неравномерность покрытия нишевых кинопроизведений в разных языковых сегментах (задокументированный пример – фильм Mike & Nick & Nick & Alice, для которого русскоязычный сегмент оказался невосполним при агрессивном сборе).
- Сформирован зафиксированный снимок корпуса от 23.04.2026 для обеспечения воспроизводимости результатов экспериментальной части.

### Работы, начатые с опережением графика (относятся к этапу 2.3)

Расширение корпуса являлось необходимым условием для проведения исследовательских работ этапа 2.2, в связи с чем часть задач следующего этапа была выполнена раньше:

- **Модуль сбора данных.** Оптимизирован мультиязычный поиск комментариев, обеспечена устойчивость работы с YouTube Data API v3 и TMDB API. Подготовлен воспроизводимый скрипт `fill_thesis_corpus.py`, позволяющий расширять корпус и пополнять слабые ячейки существующих записей. Корпус расширен с 4 393 до 19 155 комментариев, с 11 до 16 фильмов. Добавлены 5 фильмов 2025 года: Mickey 17, Sinners, Thunderbolts*, Lilo & Stitch, Mission: Impossible – The Final Reckoning. Языковой баланс выровнен: доля испанского сегмента возросла с 14 % до 34 %.
- **Модуль предобработки.** Выполнена нормализация диакритических знаков при сопоставлении стоп-слов, дополнен словарь стоп-слов специфичной лексикой обзоров YouTube для трёх языков.
- **Пользовательский интерфейс.** Разработана и интегрирована новая вкладка «Межъязыковые различия» с пятью информационными карточками (относительный фаворит каждого языка, наиболее спорный фильм, наибольшее согласие между языками, наиболее одобряемый в среднем) и сводной таблицей с настраиваемым порогом минимального числа комментариев на язык (N ≥ 50 по умолчанию, опции 75, 100, 150). Реализован бэкенд-эндпоинт `GET /api/sentiment/language-divergence`. Доработаны переводы интерфейса на трёх языках (расширение словаря i18n до ~85 ключей). Выполнены визуальные улучшения дашборда: переработаны KPI-панели, скорректированы заголовки для большей информативности.

### Этап 2.3. Доработка системы и тестирование (04.05 – 15.05)

#### Функциональное и интеграционное тестирование

- Проведено систематическое тестирование REST API: проверены все 19 эндпоинтов на корректность ответов, в том числе на граничные значения параметров (`limit`, `min_count`, неподдерживаемые значения `lang` и `sentiment`, пустые параметры).
- Выполнено визуальное тестирование клиентской части: пройдены все семь вкладок дашборда, проверено переключение языка интерфейса (RU/ES/EN), отзывчивость фильтров и корректность отображения данных.
- Идентифицированы и устранены 4 дефекта:
  - Отсутствие верхнего ограничения параметра `limit` в эндпоинтах `/api/comments` и `/api/wordcloud` (в худшем случае ответ достигал 8,7 МБ и 524 КБ соответственно). Введены ограничения `le=1000` и `le=500`.
  - Молчаливое принятие неподдерживаемых значений параметра `lang` в эндпоинте `/api/comments` (возвращался пустой результат вместо явной ошибки). Добавлена валидация со списком допустимых значений и диагностическим сообщением.
  - Битая ссылка на `logo192.png` в `manifest.json` клиентской части (предупреждение в консоли браузера). Файл манифеста приведён в соответствие с фактическими ресурсами проекта.
  - Регрессионный дефект: после введения ограничения `le=200` для `/api/comments` сломалась вкладка «Динамика», запрашивающая по 500 комментариев на язык для построения временного ряда. Граница пересмотрена до `le=1000` с сохранением защиты от чрезмерных запросов.

#### Доработка модуля сбора и обработки данных

- Усилена валидация параметров REST API: явный отказ с информативным сообщением при некорректных значениях вместо тихого возврата пустого результата. Это улучшает диагностируемость и поведение системы для интеграторов.
- Добавлен бэкенд-эндпоинт `GET /api/trends/dynamics` с серверной агрегацией временных рядов (по дню, неделе, месяцу, году). Эндпоинт заменяет прежнюю клиентскую логику, требовавшую загрузки до 1 500 сырых комментариев для построения графика динамики, что в перспективе обеспечивает масштабируемость при росте корпуса.

#### Доработка пользовательского интерфейса

- Полная переработка вкладки «Динамика» с ориентацией на использование в защите ВКР:
  - Заменены относительные временные фильтры («последние N дней») на абсолютные (выбор конкретного года). Обоснование: относительные фильтры зависят от текущей даты и не воспроизводимы между моментом написания работы и моментом защиты; абсолютные фильтры дают идентичный результат при любом обращении к замороженному корпусу.
  - Удалена гранулярность «неделя» по итогам тестирования: при четырёхлетнем диапазоне корпуса недельная гранулярность создавала ~145 точек данных с высокой шумностью и низкой интерпретируемостью.
  - Применены оптимизации производительности отрисовки: мемоизация вычислений (`useMemo`), отключение анимаций перерисовки (`isAnimationActive=false`, `animationDuration=0`), условное отображение точек графика при большом количестве данных. Устранён ранее наблюдавшийся лаг при наведении курсора.
- Упрощён компонент `FilterPanel`: удалён фильтр периода «7/30/90 дней» из вкладок «Тренды», «Тональность» и «Комментарии». Обоснование: (а) при замороженном на 23.04.2026 корпусе относительные фильтры возвращали неполные или пустые результаты, (б) функциональность временного анализа теперь полнее реализована во вкладке «Динамика» с абсолютными фильтрами, (в) сокращение количества элементов управления снижает когнитивную нагрузку пользователя.

#### Подготовка документации к ВКР

- Подготовлены внутренние документы (хранятся локально, без публикации в репозитории) для использования при написании текста ВКР:
  - Описание архитектуры программной системы с диаграммой пятислойной структуры (коллекторы → NLP-конвейер → реляционная база данных → REST API → клиентское приложение) и спецификацией модулей.
  - Руководство пользователя с описанием семи вкладок дашборда, трёх типичных сценариев использования и задокументированных ограничений системы.
- Расширен файл `README.md` репозитория: добавлен раздел «Основные функции» с перечнем вкладок интерфейса, актуализирована таблица эндпоинтов, исправлена структура проекта в соответствии с фактическим состоянием кодовой базы, заменён жёстко прописанный пароль БД на плейсхолдер.
